## 模块概述

这四个文件定义了 DeepChoice 的"骨架"——数据怎么流、检索器怎么接。全部 Agent 都依赖 `ResearchState` 和 `TaskConfig` 作为数据契约。`BaseRetriever` 是 6 路检索器的统一接口，用了模板方法模式。

在 LangGraph pipeline 中的位置：
```
TaskConfig (输入) → ResearchState (流转) → 6路检索器 → Agent管道
```

## 一、state.py — 研究状态定义

### 完整代码

```python
from typing import TypedDict

class ResearchState(TypedDict):
    task: dict
    sub_questions: list[str]
    search_results: list[dict]
    source_scores: list[dict]
    conflicts: list[dict]
    evidence_chains: list[dict]
    report: str
    confidence: str
    knowledge_gaps: list[str]
    retry_count: int
    partial_failures: list[str]
    current_phase: str
```

### 逐字段解读

| 字段 | 类型 | 谁写入 | 谁读取 | 用途 |
|------|------|--------|--------|------|
| `task` | dict | Orchestrator | 所有 Agent | 用户原始 query + scene_context + constraints |
| `sub_questions` | list[str] | QueryAnalyzer | MultiRetriever / SelfReviewer | 拆分后的 5 个研究子问题 |
| `search_results` | list[dict] | MultiRetriever | SourceEvaluator / ReportGenerator | 6 路检索器原始结果 |
| `source_scores` | list[dict] | SourceEvaluator | ConflictDetector / EvidenceChain | 每条结果的可信度评分 |
| `conflicts` | list[dict] | ConflictDetector | EvidenceChain / ReportGenerator | 检测到的信息冲突 |
| `evidence_chains` | list[dict] | EvidenceChain | ReportGenerator / SelfReviewer | 按证据强度组织的结论链 |
| `report` | str | ReportGenerator | SelfReviewer | 最终研究报告文本 |
| `confidence` | str | SelfReviewer | Orchestrator (路由) | high / medium / low |
| `knowledge_gaps` | list[str] | SelfReviewer | Orchestrator (路由) | 补搜索的关键词列表 |
| `retry_count` | int | Orchestrator | Orchestrator / SelfReviewer | 重试次数，>=1 强制结束 |
| `partial_failures` | list[str] | MultiRetriever | ReportGenerator | 故障检索器名称列表 |
| `current_phase` | str | Orchestrator | — | 当前管道阶段标识 |

### 设计决策

**1. 为什么用 TypedDict 而不是 Pydantic BaseModel？**

LangGraph 的 StateGraph 要求 state 是 dict 类型。TypedDict 提供类型提示但不强制运行时校验——LangGraph 内部会对 state 做 merge 操作，如果用 Pydantic 强校验，节点返回部分字段时可能抛异常。实际上所有 Agent 的 `run()` 返回的都是 `dict`，LangGraph 自动 merge 进 ResearchState。

**2. 为什么 search_results 是 `list[dict]` 而不是强类型？**

6 路检索器返回的结果结构不完全一样。Tavily 有 `url`/`title`/`snippet`，Chroma 有 `content`/`metadata`/`distance`。用 `list[dict]` 保持灵活，实际校验下沉到 SourceEvaluator，它通过 `result.get("url", "")` 安全读取，不存在就返回空字符串。

**3. partial_failures 为什么设计成独立字段而不是嵌在 search_results 里？**

分离关注点。`search_results` 是"可用的数据"，`partial_failures` 是"元信息"——下游可以只看 search_results 正常处理，ReportGenerator 额外消费 partial_failures 生成数据覆盖声明。两者不混在一起，消费者不用从成功结果里过滤失败记录。

## 二、task.py — 任务配置与校验

### 完整代码

```python
import json
from pathlib import Path
from pydantic import BaseModel, Field

class TaskConfig(BaseModel):
    query: str
    scene_context: str = "team"
    constraints: list[str] = Field(default_factory=list)
    report_format: str = "what_why_how"

    def model_post_init(self, __context):
        valid_scenes = {"solo", "team", "enterprise"}
        if self.scene_context not in valid_scenes:
            raise ValueError(f"scene_context must be one of {valid_scenes}")
        valid_formats = {"what_why_how", "evidence_first", "comparison_matrix"}
        if self.report_format not in valid_formats:
            raise ValueError(f"report_format must be one of {valid_formats}")

def load_task(path: str) -> TaskConfig:
    raw = json.loads(Path(path).read_text(encoding="utf-8"))
    return TaskConfig(**raw)
```

### 字段解读

- **query**: 必填，用户原始问题，如 "React vs Vue 技术选型"
- **scene_context**: 场景：solo（个人）/ team（团队）/ enterprise（企业）。默认 team。决定 QueryAnalyzer 的分解策略和 ReportGenerator 的建议偏向
- **constraints**: 约束条件列表，如 `["预算<500/月", "需要中文文档"]`
- **report_format**: 输出格式，what_why_how / evidence_first / comparison_matrix，默认 what_why_how

### 设计决策

**1. 为什么用 Pydantic 而不是 dataclass？**

`model_post_init` 提供了"构造后自动执行校验"的钩子。如果用 dataclass，校验逻辑要写在 `__post_init__` 里，或者分散在外部。Pydantic 的 Field + model_post_init 让"默认值定义"和"校验规则"都在同一个类里，可读性好。

而且 `load_task` 里 `TaskConfig(**raw)` 一行就能把 JSON dict 转成校验过的对象——Pydantic 自动做了类型转换和必填检查。

**2. scene_context 默认值为什么是 "team" 而不是 "solo" 或 None？**

默认值的选择原则：选"最常见的场景"而不是"最保守的"。团队场景是技术选型最高频的使用场景（个人学习不需要系统化比较，企业级有专门架构师），设为默认值减少澄清轮次。

**3. model_post_init 校验 vs Pydantic validator 装饰器，为什么选前者？**

`model_post_init` 在 `__init__` 完成后执行一次，适合做跨字段校验（虽然当前没有跨字段逻辑）。validator 更适合单字段转换。这个选择没有对错，只是风格偏好——把所有校验集中在一个方法里而不是分散在多个 validator。

## 三、retrievers/base.py — 检索器模板方法

### 完整代码

```python
import time

class BaseRetriever:
    source: str = "base"

    async def search(self, query: str, sub_questions: list[str], max_results: int = 7) -> dict:
        t0 = time.monotonic()
        try:
            results = await self._do_search(query, sub_questions, max_results)
            return {
                "source": self.source,
                "status": "success",
                "results": results,
                "error": None,
                "latency_ms": round((time.monotonic() - t0) * 1000),
            }
        except Exception as e:
            return {
                "source": self.source,
                "status": "failed",
                "results": [],
                "error": str(e),
                "latency_ms": round((time.monotonic() - t0) * 1000),
            }

    async def _do_search(self, query, sub_questions, max_results) -> list[dict]:
        raise NotImplementedError
```

### 设计决策

**1. 为什么用模板方法模式（Template Method）？**

`search()` 定义了统一的外壳：计时 → 调子类实现 → 成功包装成统一格式 → 失败也包装成统一格式（不抛异常）。6 个子类（Tavily / Chroma / GitHub / Arxiv / Community / Official）只需实现 `_do_search()`，不必关心计时、错误包装、返回格式统一这些重复逻辑。

这就是 **Template Method 模式**：父类定义算法骨架（search），子类实现具体步骤（_do_search）。

**2. 为什么用 `time.monotonic()` 而不是 `time.time()`？**

`time.time()` 受系统时间调整影响（NTP 校时、用户手动改时间），可能导致 latency 为负数。`time.monotonic()` 是单调递增的，不受系统时间变化影响，只适合测量时间间隔。这是一个工程细节，面试时提出来说明你注意了边界情况。

**3. 为什么失败不抛异常，而是返回 `{"status": "failed", ...}`？**

回到 MultiRetriever 的 `asyncio.gather(return_exceptions=True)` 场景：如果这里抛异常，gather 会把异常当返回值，MultiRetriever 需要区分"哪种 Exception 对应哪个检索器"。返回统一 dict 让异常处理变得一致——检索器内部自己能处理的就自己处理（网络超时 → status: failed），处理不了的才让 Exception 透传。

**4. source 是类属性，不同子类覆盖它。这利用了 Python 什么机制？**

类属性继承。子类定义 `source = "tavily"` 会覆盖父类的 `source = "base"`。实例访问 `self.source` 时，Python 先找实例属性 → 再找类属性 → 再找父类属性（MRO 顺序）。

### 盲区补课：模板方法模式

这是 GoF 23 种设计模式之一。结构：
```
父类:  定义算法骨架 (template method)，调用抽象步骤
子类:  实现具体步骤
```

本文件的结构：
```
BaseRetriever.search()     ← 模板方法（计时 → 搜索 → 包装）
BaseRetriever._do_search() ← 抽象步骤（raise NotImplementedError）
TavilySearch._do_search()  ← 具体实现
```

**为什么不用 @abstractmethod？** 因为你可能想写一个 MockRetriever 做测试，不强制实现抽象方法。`raise NotImplementedError` 更灵活——不实现也能实例化，调用时才报错。

## 四、retrievers/__init__.py — 检索器注册表

### 完整代码

```python
from .tavily_search import TavilySearch
from .chroma_kb import ChromaKB
from .github_api import GitHubSearch
from .arxiv_api import ArxivSearch
from .community import CommunitySearch
from .official import OfficialSearch

RETRIEVER_REGISTRY = {
    "tavily": TavilySearch,
    "chroma": ChromaKB,
    "github": GitHubSearch,
    "arxiv": ArxivSearch,
    "community": CommunitySearch,
    "official": OfficialSearch,
}

__all__ = [
    "TavilySearch", "ChromaKB", "GitHubSearch",
    "ArxivSearch", "CommunitySearch", "OfficialSearch",
    "RETRIEVER_REGISTRY",
]
```

### 设计决策

**1. 为什么用 dict 注册表而不是硬编码 import？**

MultiRetriever 用 `for name, cls in RETRIEVER_REGISTRY.items()` 就能遍历所有检索器。新增检索器只需：① 写一个新类继承 BaseRetriever ② 在 `__init__.py` 注册一行。MultiRetriever 代码完全不需要改。

这就是 **开闭原则（Open-Closed Principle）**：对扩展开放（加检索器），对修改关闭（不改 MultiRetriever）。

**2. 这个模式在工程里叫什么？**

**注册表模式（Registry Pattern）** 或 **插件架构（Plugin Architecture）**。核心思想是"用数据（dict）代替代码（if-else）来描述有哪些组件可用"。

面试可以说："6 路检索器通过注册表模式统一接入，新增检索器只需注册一行，MultiRetriever 零改动。"

## 面试拷打

### Q1: ResearchState 为什么不用 Pydantic？
TypedDict 是 LangGraph 兼容性选择。LangGraph 节点返回 dict，框架自动 merge 进 state。Pydantic 的强校验在这个场景下反而碍事——节点可能只返回部分字段。

### Q2: BaseRetriever 的 search 方法为什么自己 try-catch，而不是让调用方处理？
因为 6 路检索器并行执行（asyncio.gather），一路挂了不应该影响其他路。把异常处理下沉到检索器层，调用方（MultiRetriever）不用为每种检索器写不同的异常处理。

### Q3: 如果要新增第 7 路检索器，需要改哪些文件？
- 新建 `retrievers/xxx.py`，继承 `BaseRetriever`，实现 `_do_search`
- `retrievers/__init__.py` 加一行 import + dict 注册
- 其他文件零改动

### Q4: `time.monotonic()` 和 `time.time()` 的区别？
- `time.time()`: 返回 Unix 时间戳，受系统时钟调整影响
- `time.monotonic()`: 单调递增计时器，只适合测时间间隔
- `time.perf_counter()`: 精度最高的计时器（包含 sleep 时间），benchmark 首选
- 本文件用 monotonic 是正确的——只需要"请求花了多少毫秒"
