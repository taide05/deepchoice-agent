## 模块概述

澄清模块是 DeepChoice 的"前置交互层"——在启动研究管道之前，通过多轮对话帮用户把模糊问题说清楚。输出格式是管道的终点——把研究结果渲染成可读的报告。

```
用户输入 → SessionManager (会话 + 状态管理)
         → ClarificationAgent (多轮对话 + 技术推荐)
         → 确认 → 启动 Orchestrator 管道
         → ReportGenerator → 格式渲染器 → 最终报告
```

## 一、session_manager.py — 会话与状态管理

### SessionState — 会话状态模型

```python
class SessionState(BaseModel):
    session_id: str = ""
    status: Literal["clarifying", "ready", "running", "done"] = "clarifying"

    # 用户需求三要素（必填）
    candidate_techs: list[str] = []           # 候选技术
    scene: Literal["solo", "team", "enterprise"] | None = None
    complexity: Literal["simple", "medium", "complex"] | None = None

    # 附加信息
    constraints: list[str] = []
    unknown_techs: bool = False               # 用户不懂技术，需要推荐
    tech_recommendations: list[dict] = []

    # 澄清进度
    clarify_rounds: int = 0
    filled_required: list[str] = []
    missing_required: list[str] = []
    clarity_score: float = 0.0               # 0-1，衡量收集了多少必填项

    # 对话记录
    messages: list[dict] = []

    # 输出
    clarified_task: dict | None = None
    sub_questions: list[str] | None = None

    last_active: float = Field(default_factory=time.time)
```

**状态字段设计逻辑：**

三个必填项（candidate_techs / scene / complexity）是澄清的**最小信息集**。clarity_score 按 `已填项数 / 3` 计算——填了 1 项 = 0.33，全填 = 1.0。这是**可量化的澄清进度**，前端可以根据这个分数显示进度条。

### SessionManager — 增删查 + 软门禁

```python
class SessionManager:
    SESSION_TIMEOUT = 1800  # 30分钟过期
    KNOWN_TECHS = {
        "python", "javascript", "typescript", ..., "streamlit", "celery", ...
    }  # 100+ 已知技术关键词

    def __init__(self):
        self._sessions = {}  # 内存存储，dict[session_id] = SessionState

    def create(self, query: str) -> dict:
        """创建新会话，从 query 中提取技术关键词"""
        session_id = f"clarify_{uuid4().hex[:12]}"
        state = self._extract_initial_state(query)
        self._sessions[session_id] = state
        state.session_id = session_id
        return self._response(state)

    def process_message(self, session_id: str, message: str) -> dict:
        """处理用户消息，更新状态"""
        state = self._get_or_raise(session_id)
        state.last_active = time.time()
        state.messages.append({"role": "user", "content": message})
        state.clarify_rounds += 1
        self._update_state_from_message(state, message)
        return self._response(state)

    def finalize(self, session_id: str) -> dict:
        """结束澄清，应用软门禁"""
        state = self._get_or_raise(session_id)
        self._apply_soft_gate(state)
        state.status = "ready"
        return self._response(state)
```

### 设计决策

**1. 为什么用内存 dict 存储而不是数据库？**

澄清会话是短暂的（最多 3 轮对话、几分钟生命周期）。30 分钟超时后自动清理。不需要持久化——用户刷新页面就重新开始。这符合"最短生命周期存储"原则：活多久就用多重的存储。

**2. 为什么 `KNOWN_TECHS` 是 `set` 而不是 `list`？**

`if keyword in KNOWN_TECHS` 在 set 中是 O(1)，在 list 中是 O(n)。100+ 关键词，每次 `_detect_tech_keywords` 都要遍历全部，用 set 节省微不足道的时间——但这是一种**习惯性优化**，面试时可以提。

**3. 软门禁（Soft Gate）的设计意图：**

```python
def _apply_soft_gate(self, state: SessionState) -> SessionState:
    if state.scene is None:
        state.scene = "team"       # 默认团队场景
    if state.complexity is None:
        state.complexity = "medium" # 默认中等复杂度
    # 如果 candidate_techs 仍为空，不填默认值——这是最核心的必填项
```

"软门禁"的意思是：不是所有必填项都必须用户填写，不填的给合理默认值。唯一不能默认的是 candidate_techs——不知道用户要比较什么技术，无法替用户决定。

**4. SessionManager 为什么不自己调 LLM？**

单一职责。SessionManager 管理状态生命周期（创建/查询/更新/过期/清理），ClarificationAgent 负责对话逻辑（什么时候问什么问题）。分开了以后，如果未来要换对话策略（比如从多轮对话改成表单），SessionManager 不用改。

### 盲区补课：软门禁 vs 硬门禁

| | 硬门禁（Hard Gate） | 软门禁（Soft Gate） |
|------|------|------|
| 行为 | 不填完→不让过 | 不填→给默认值再过 |
| 适用 | 信息缺失会导致错误结果 | 信息缺失影响精度但不致命 |
| 本项目中 | `_update_state_from_message`（未实现） | `_apply_soft_gate`（当前策略） |

技术选型场景中，不知道场景（solo/team/enterprise）只会影响建议的偏向，不会导致错误推荐——所以用软门禁。如果换成医疗诊断场景，必须用硬门禁。

**设计决策：推荐库为什么放代码里而不是数据库/配置文件？**

推荐数据量小（6 个分类 × 5-7 个技术 = ~35 条），更新频率低（一个月改一次），放代码里最直接。这遵循"数据离使用点越近越好"的原则——不引入数据库依赖做一件不需要数据库的事。规模大了（几百个分类、几千条目）再迁移到向量数据库。

### CATEGORY_KEYWORDS — 意图识别

```python
CATEGORY_KEYWORDS: dict[str, str] = {
    "前端": "fc14a2e9", "网站": "fc14a2e9", "网页": "fc14a2e9", "ui": "fc14a2e9",
    "后台": "a7b31d5f", "后端": "a7b31d5f", "api": "a7b31d5f", ...
    "对话": "b82c4e6a", "agent": "b82c4e6a", "ai": "b82c4e6a", ...
    ...
}

def _match_categories(query: str) -> list[str]:
    query_lower = query.lower()
    matched = []
    for kw, cat_id in CATEGORY_KEYWORDS.items():
        if kw in query_lower and cat_id not in matched:
            matched.append(cat_id)
    return matched if matched else ["b82c4e6a"]  # 默认 AI 分类
```

**设计决策：关键词匹配做意图识别——为什么不用 LLM？**

意图识别太简单了，不值得调 LLM。"我想比较前端框架"——关键词"前端"就在 CATEGORY_KEYWORDS 里，100% 命中。LLM 的延迟和成本在这里没有收益。

**默认 fallback 是 "b82c4e6a"（AI/Agent 框架）而不是别的分类——为什么？** 因为 DeepChoice 是技术选型工具，用户最可能比较的就是 AI 相关技术。

### ClarificationAgent — 三行动决策

```python
class ClarificationAgent:
    def _decide_action(self, state: SessionState) -> str:
        if state.clarify_rounds >= 3:
            return "confirm"                                    # 规则1: 最多3轮
        if not state.missing_required:
            return "confirm"                                    # 规则2: 信息已齐全
        if "candidate_techs" in state.missing_required and state.unknown_techs:
            return "recommend"                                  # 规则3: 用户不懂技术→推荐
        return "ask"                                            # 规则4: 默认→追问
```

**决策树：**

```
轮次 >= 3?                   → confirm（强制结束）
    ↓ No
missing_required 为空?       → confirm（信息齐全）
    ↓ No
用户不懂技术 + 缺少候选技术? → recommend（推荐模式）
    ↓ No
                              → ask（追问模式）
```

**三个 handle 方法：**

| 方法 | 触发条件 | LLM 调用 | 行为 |
|------|---------|---------|------|
| `_handle_ask` | 有缺口，用户懂技术 | 是 | LLM 根据 prompt 生成下一个问题 |
| `_handle_recommend` | 用户不懂技术 | 是 | LLM 生成引导语 + 静态推荐列表（前端卡片渲染） |
| `_handle_confirm` | 信息齐全或轮次到上限 | 是 | LLM 生成需求摘要，确认后启动研究 |

**设计决策：为什么只在 `not state.scene` 时才更新？**

用户可能在前一轮对话中已经说了场景（"我们团队有 30 个人"），LLM 在第二轮仍然输出 scene 字段。如果不加 `not state.scene` 判断，第二轮可能覆盖第一轮的正确值。这是**防御性状态管理**——已确认的信息不允许被覆盖。

### 输出结构

生成的 Markdown 报告分四段：
1. **五维对比矩阵**（表格：维度 | 判据 | 证据强度）
2. **技术对比详表**（表格：维度 | 技术A | 技术B | 胜出）
3. **争议与分歧**（列表：冲突点 + 仲裁结果）
4. **建议**（场景标注 + 首选依据 + 置信度 + 下一步行动）

### 设计决策

**1. 关键词归类 vs LLM 分类：** 和 SourceEvaluator 一样，规则引擎比 LLM 更适合"把结论归到五个维度"——关键词命中是可解释的、零成本的、可复现的。

**2. 五维框架的一致性：** 这个五维分类（功能/性能/生态/体验/场景）在三个地方出现：
- QueryAnalyzer 的分解 prompt
- SelfReviewer 的 checklist 第 3 项
- comparison_matrix 的维度归类

这是**贯穿全管道的统一框架**——从问题分解到质量审查到报告输出，同一个五维标准。面试时强调"全管道五维对齐"是一致性设计的体现。

**3. 中文关键词 + 英文关键词双覆盖：** 因为 6 路检索器返回的结果有中文（Tavily 国内源）也有英文（GitHub/Arxiv），维度归类需要同时处理两种语言。

### `_extract_techs` — 技术名提取

```python
def _extract_techs(query: str) -> list[str]:
    for sep in [" vs ", " VS ", " versus ", " 对比 ", " 比较 "]:
        if sep in query:
            parts = query.split(sep)
            return [p.strip() for p in parts[:2]]
    return ["方案A", "方案B"]
```

简单粗暴——从 "React vs Vue" 中提取 ["React", "Vue"]。处理中英文分隔符（vs / VS / versus / 对比 / 比较）。如果找不到分隔符，返回默认占位符。

## 面试拷打

### Q1: 澄清模块为什么不直接用表单（三个下拉框），而要做多轮对话？
多轮对话能处理"表单覆盖不到"的情况。用户说"我想做个后台管理系统，不知道用啥好"——表单怎么填？candidate_techs 填什么？对话模式下 ClarificationAgent 可以推荐候选技术，再让用户选择。表单适合"用户知道自己要什么"，对话适合"用户描述需求但不知道怎么表达"。

### Q2: 为什么 SessionManager 用 Pydantic 而 ResearchState 用 TypedDict？
SessionManager 不挂在 LangGraph 上，不需要 TypedDict 兼容性。Pydantic 提供更丰富的校验（Literal 类型限制、Field default_factory）和序列化能力，适合状态管理。

### Q3: TECH_RECOMMENDATION_MAP 的星星数（stars）会过期，怎么维护？
当前是静态快照，确实会过期。生产环境中应该接入 GitHub API 动态获取 star 数，或至少定期（每周）跑脚本更新。但面试中提出这个问题本身就是加分项——说明你注意到"静态数据的时效性"。

### Q4: 三种报告格式分别适合什么场景？
- **what_why_how**: 适合"我想了解这个技术是什么、为什么选、怎么用"——面向学习/调研
- **evidence_first**: 适合"给我证据，我自己判断"——面向技术决策者
- **comparison_matrix**: 适合"A vs B"比较——面向具体选型

### Q5: comparison_matrix 的维度归类用关键词匹配，中文"功能覆盖度"和英文"feature coverage"都能命中 "功能/feature"，但如果结论是中英混合的（"这个API的throughput很高"），会怎样？
关键词匹配不关心语言混合——`"throughput"` 在 `dim_keywords["performance"]` 中，命中了就归入性能维度。这是关键词匹配的优势——语言无关。

## 全管道面试总结

### 一句话概括 DeepChoice

"技术选型 Deep Research Agent——7 步 LangGraph 管道（分解→检索→评估→冲突检测→证据链→报告→自审查），6 路并行检索器，规则引擎 + LLM 混合架构。"

### 核心设计原则（贯穿全管道）

| 原则 | 体现 |
|------|------|
| **Fail open** | 默认 fallback 选高不选低、单路失败不阻塞、阈值宽松不严格 |
| **单一路数据不决策** | consistency 维度的交叉互验、2+ 来源才给 strong |
| **LLM 用于推理，规则用于分类** | ConflictDetector 用 LLM 仲裁（推理），SourceEvaluator 用规则打分（分类） |
| **可追溯 > 自动决策** | 每个结论有来源 URL、争议标记 disputed、数据覆盖声明 |
| **注册表模式** | 检索器、报告格式、技术推荐，新增只需注册一行 |

### 面试可能被追问的架构问题

**Q: 为什么不用单一 LLM 一次生成最终报告？**
A: 不可控、不可追溯、不可复现。管道分解后每个环节可独立调试、可量化（SourceEvaluator 分数、SelfReviewer passed_count）、可复现（相同输入→相同输出）。

**Q: 如果要把这个项目扩展到 100 路检索器，哪里会先崩？**
A: MultiRetriever 的 `asyncio.gather` 会先崩——100 个并发 HTTP 连接可能触发系统 fd 限制或目标服务器限流。需要加**信号量（Semaphore）**控制并发数。

**Q: 自审查重试最多 1 次，如果还不够呢？**
A: 当前设计选择告知局限性（报告标注 confidence: low + 列出 knowledge_gaps），让用户决定是否手动补搜索。比无限重试更诚实。
